# Plan 05-02: Phase Listing Pagination (FLOW-02)

<objective>
Add pagination support to phase listing across three locations: the MCP `mcp_list_phases` tool (offset/limit parameters), the roadmap workflow (auto-collapse completed phases + paginate at 20/page), and the progress workflow (metrics table truncation). This makes MAXSIM usable for projects with 50+ phases.
</objective>

@.planning/phases/05-Workflow-Coverage/05-RESEARCH.md
</execution_context>

<context>
**Existing code (READ before implementing):**
- `packages/cli/src/mcp/phase-tools.ts` -- Current `mcp_list_phases` tool (lines 78-122) lacks offset/limit params
- `packages/cli/src/core/phase.ts` -- `cmdPhasesList` already supports `offset`/`limit` via `PhasesListOptions`
- `packages/cli/src/core/types.ts` -- `PhasesListOptions` type already has `offset?: number` and `limit?: number`
- `packages/cli/src/cli.ts` -- CLI handler already parses `--offset`/`--limit` flags
- `templates/workflows/roadmap.md` -- Current roadmap display workflow (no pagination)
- `templates/workflows/progress.md` -- Current progress workflow with metrics display

**Key insight from RESEARCH.md:** The core TypeScript function `cmdPhasesList` already implements pagination. The CLI handler already parses the flags. Only the MCP tool needs updating to expose these parameters. The roadmap and progress workflows are pure markdown changes.

**Locked decisions:**
- Page size: 20 phases
- Roadmap UX: auto-collapse completed + paginate remaining
- Pagination scope: MCP tool + roadmap + metrics table
- Only paginate when total > 20 (no "Page 1 of 1" for small projects)
</context>

<task id="01" type="auto">
<name>Add pagination to MCP list-phases tool</name>
<files>
- packages/cli/src/mcp/phase-tools.ts

<action>
Update the `mcp_list_phases` MCP tool in `packages/cli/src/mcp/phase-tools.ts` to add pagination parameters.

**Changes to the existing tool (lines 78-122):**

1. Add two new Zod parameters to the tool schema:
   - `offset: z.number().optional().default(0).describe('Number of phases to skip (for pagination)')`
   - `limit: z.number().optional().default(20).describe('Maximum number of phases to return')`

2. Update the handler function signature to destructure `offset` and `limit` from params.

3. After sorting `dirs`, apply pagination:
   - Store `const total_count = dirs.length;`
   - Slice: `const paginated = dirs.slice(offset, offset + limit);`
   - Calculate: `const has_more = offset + limit < total_count;`

4. Update the return value to include pagination metadata:
   - Change `{ directories: dirs, count: dirs.length }` to `{ directories: paginated, count: paginated.length, total_count, offset, limit, has_more }`
   - Update the summary string to: `Showing ${paginated.length} of ${total_count} phase(s)`

5. Also update the empty-directory early return to include pagination fields:
   - `{ directories: [], count: 0, total_count: 0, offset, limit, has_more: false }`

Do NOT change the tool name, description format, or error handling pattern. Only add parameters and update the response shape.

**Reference:** See 05-RESEARCH.md "Example 1: MCP Tool with Pagination Parameters" for the exact target code.
</action>

<action>
Update two workflow markdown files to add pagination display logic.

**1. roadmap.md -- Auto-collapse completed phases + paginate:**

Modify the `render` step (currently lines 36-72) to add two behaviors:

**Auto-collapse completed phases:**
After the milestone header, render phases in two visual groups:
- Completed phases as collapsed one-liners: `checkmark Phase {number}: {name}` (no plan counts, no status label -- just checkmark + name)
- Active/upcoming phases with full detail (current format with icon, label, plan counts)

This applies regardless of phase count (always useful for visual clarity).

**Pagination (only when total phases > 20):**
After rendering all phases, if the total phase count exceeds 20:
- Show only phases for the current "page" (default page 1, 20 per page)
- The workflow should accept a `--page N` argument from the command invocation
- Add footer line: `Showing phases {first}-{last} of {total}. Use --page N for next.`
- If total is 20 or fewer, show all phases without any pagination footer

Update the `check_planning` step to parse a `--page` argument if provided. Default to page 1.

Update the success_criteria section to include the new behaviors.

**2. progress.md -- Metrics table truncation:**

In the `report` step, add a note that when displaying the performance metrics table from STATE.md:
- Show only the last 20 entries (most recent) by default
- If there are more than 20 entries, add a note: `Showing last 20 of {total} metrics entries.`
- This is a display-time truncation only -- STATE.md retains all metrics as the source of truth

This is a small addition to the report step's instructions for how to render the metrics section. Do NOT restructure the existing workflow -- just add the truncation instruction to the relevant step.
</action>
