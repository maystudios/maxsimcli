<objective>
Create the drift-checker agent prompt, the `/maxsim:check-drift` command, and the check-drift workflow that together enable users to run a single command to compare their `.planning/` spec against the actual codebase and produce a structured DRIFT-REPORT.md.
</objective>

<execution_context>
Plan 01 (Wave 1) created the drift core module with CLI tool commands (drift read-report, drift extract-requirements, etc.), types, and init context. This plan creates the three-layer stack (command -> workflow -> agent) that uses those tools.

Key user decisions to honor:
- Full inventory (aligned + drifted items), not just mismatches
- Phase-level summary at top, per-requirement detail below
- Three severity tiers: critical / warning / info
- Evidence per mismatch (spec line + code location)
- Per-item fix recommendations
- Two directional sections: "Spec ahead of code" and "Code ahead of spec"
- Dedicated "Undocumented Features" section
- Per-criterion breakdown for partial implementations
- Timestamped with diff tracking (NEW/RESOLVED/UNCHANGED vs previous report)
- YAML frontmatter for machine consumption
- Partial check on incomplete spec (warn about missing pieces, don't refuse)
- Full .planning/ sweep (REQUIREMENTS, STATE decisions, CONVENTIONS, NO-GOS, phase summaries)
- All milestones including archived (archived regressions in separate section with lower severity)
- Descoped/deferred items excluded
- No-go violation checking (critical severity)
- Implementation = source code + tests
- Convention compliance: best-effort pattern check (info-level with confidence)
- Fresh codebase scan every run
- Verify phase summaries against actual code
- The drift-checker agent should NOT be interactive -- it produces a complete report
- Do NOT store drift state in STATE.md (DRIFT-REPORT.md is single source of truth)
</execution_context>

**1. Create `templates/agents/maxsim-drift-checker.md`** -- the drift analysis agent prompt.

Use the exact agent prompt structure established in Phase 3 (system map, role, upstream/downstream contracts, input validation, needs frontmatter). Reference `maxsim-verifier.md` for the structural template.

The agent frontmatter:
```yaml
---
name: maxsim-drift-checker
description: Compares .planning/ spec against codebase state, producing DRIFT-REPORT.md with severity-tiered findings and fix recommendations.
tools: Read, Write, Bash, Grep, Glob
color: yellow
needs: [requirements, roadmap, state, nogos, conventions, codebase_docs]
---
```

Agent system map: Include ALL 15 agents (14 existing + drift-checker itself). Copy the system map format from any Phase 3 agent.

Role section: The drift-checker reads all spec files (.planning/REQUIREMENTS.md, ROADMAP.md, STATE.md, NO-GOS.md, CONVENTIONS.md, phase SUMMARYs), then systematically analyzes the codebase for evidence of implementation.

Multi-pass execution protocol (critical for avoiding context overload):
- **Pass 1 -- Spec Extraction:** Use CLI tools to extract structured data:
  - `node maxsim-tools.cjs drift extract-requirements` -- get all requirements with completion status
  - `node maxsim-tools.cjs drift extract-nogos` -- get all no-go rules
  - `node maxsim-tools.cjs drift extract-conventions` -- get convention text
  - `node maxsim-tools.cjs roadmap analyze` -- get phase structure and success criteria
  - Read STATE.md decisions section directly
  - Read phase SUMMARY.md files for completed phase claims
- **Pass 2 -- Codebase Analysis:** For EACH requirement/criterion:
  - Use Grep and Glob to search for evidence of implementation
  - Check for test files related to the feature
  - Record file paths, line numbers, and relevant code snippets as evidence
  - Classify: aligned (both spec and code agree), spec_ahead (spec says done but code missing/incomplete), code_ahead (code exists but spec doesn't mention)
- **Pass 3 -- No-Go and Convention Check:**
  - For each no-go rule, search codebase for violations using Grep
  - For conventions, do best-effort pattern matching (info-level, include confidence indicator)
- **Pass 4 -- Archived Phase Regression Check:**
  - Read archived phase directories
  - Cross-reference archived requirements with current milestone -- skip if superseded
  - Report findings in separate section with info severity (unless no-go violation)
- **Pass 5 -- Synthesis:** Compile all findings into DRIFT-REPORT.md:
  - If previous report exists (`node maxsim-tools.cjs drift previous-hash`), compare items to determine NEW/RESOLVED/UNCHANGED labels
  - Write report with full YAML frontmatter
  - Use `node maxsim-tools.cjs drift write-report --content-file <tmpfile>` to write the report

Report format: Follow the exact structure from the research (Code Example 5). Include:
- YAML frontmatter (status, checked timestamp, counts, spec_files_checked, previous_hash)
- Phase Overview table
- "Spec Ahead of Code" section with CRITICAL / WARNING / INFO subsections
- "Code Ahead of Spec" section with UNDOCUMENTED subsection
- "No-Go Violations" section
- "Convention Compliance" section
- "Archived Phase Regressions" section (if any)
- "Diff Summary" table (NEW/RESOLVED/UNCHANGED counts)

Severity classification rules (per user decisions):
- CRITICAL: Requirement marked `[x]` complete but feature code completely missing. No-go violation found in code.
- WARNING: Requirement partially implemented (some criteria pass, others fail). Feature code exists but tests missing.
- INFO: Convention compliance findings (with confidence indicator). Naming/cosmetic drift. Archived regressions (unless no-go).

Exclusion rules:
- Items in REQUIREMENTS.md "Out of Scope" section: skip silently
- Items explicitly deferred in CONTEXT.md or STATE.md: skip silently
- Requirements not yet marked complete (`[ ]`): report as "not yet started" in aligned section, not as drift

Upstream contract: Receives CheckDriftContext from workflow (init check-drift output). Must validate that has_planning is true and at least has_requirements or has_roadmap is true.

Downstream contract: Writes DRIFT-REPORT.md directly to `.planning/`. Returns summary to workflow (status, counts only -- not full report content to avoid context bloat).

Input validation: If no `.planning/` directory exists, output error message. If spec files are partially missing, run check on whatever exists and warn at top of report.

**2. Update `templates/agents/AGENTS.md`:** Add the drift-checker to the agent registry table. Follow the exact format used for other agents. Add it after the integration-checker entry.

- Post-completion offer: "Realign to code, to spec, or done?" -- if user chooses realign, invoke `/maxsim:realign` with the appropriate direction

**2. Create `templates/workflows/check-drift.md`** -- the orchestration workflow.

Follow the pattern from `map-codebase.md` workflow. Structure:

Step 1 -- Init: Call `node ~/.claude/maxsim/bin/maxsim-tools.cjs init check-drift` to get CheckDriftContext. Parse the JSON result to understand what spec files exist and what the project state is.

Step 2 -- Validation: Verify `has_planning` is true. If not, tell user to run `/maxsim:new-project` first. Verify at least one spec file exists (has_requirements or has_roadmap).

Step 3 -- Spawn drift-checker agent: Use the Task tool to spawn `maxsim-drift-checker` agent. Pass the CheckDriftContext as the agent's context. The agent will:
- Read all spec files
- Scan the codebase systematically
- Write DRIFT-REPORT.md to `.planning/`
- Return a summary (status, counts)

Step 4 -- Report results: After agent completes, read the drift report summary. Display key findings to user:
- Overall status (aligned or drift detected)
- Critical/warning/info counts
- Brief list of critical findings (if any)

Step 5 -- Offer realignment: Present the user with options:
- "Realign to code" -- updates `.planning/` to match codebase (invoke `/maxsim:realign to-code`)
- "Realign to spec" -- creates new phases for code gaps (invoke `/maxsim:realign to-spec`)
- "Done" -- no action needed

Include the model resolution from CheckDriftContext (drift_model field) for the Task tool model parameter.

The workflow must NOT read the full DRIFT-REPORT.md content into its own context after the agent writes it -- only read the frontmatter for counts. This prevents context bloat per the research recommendations.
</action>
