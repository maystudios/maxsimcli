<action>
Rewrite `templates/references/questioning.md` to include all of the following (see 02-RESEARCH.md Pattern 1 and Pattern 2 for exact structure):

1. **Keep existing sections intact:** Preserve the existing `<philosophy>`, `<how_to_question>`, and `<anti_patterns>` sections. Only replace the `<context_checklist>` section.

2. **Replace `<context_checklist>` with `<domain_checklist>`:** Create a comprehensive checklist with 4 categories and 20+ domains:
   - **Core (6):** Auth approach, Data model, API style, Deployment target, Error handling strategy, Testing strategy
   - **Infrastructure (5):** Caching strategy, Search, Monitoring/logging, CI/CD, Environments
   - **UX/Product (5):** User roles/permissions, Notifications, File uploads, Internationalization, Accessibility
   - **Scale/Ops (5):** Performance targets, Concurrency model, Data migration, Backup/recovery, Rate limiting

   Each domain should have parenthetical examples of options (as shown in 02-RESEARCH.md Pattern 1).

3. **Add gate logic section:** Include explicit rules:
   - AI internally counts rounds (1 round = 1 AskUserQuestion call)
   - "Ready?" appears ONLY when round >= 10 AND coverage >= 80%
   - Coverage formula: covered_count / (total - na_count) >= 0.80
   - Before "Ready?", display domain coverage summary by category
   - Mark domains as N/A generously -- include decision tree examples per project type

4. **Add `<nogos_tracking>` section** (see 02-RESEARCH.md Pattern 2):
   - During questioning: watch for rejection signals (explicit rejections, past failures, strong opinions, anti-patterns)
   - Silently accumulate, do NOT confirm each one individually
   - Challenge-based probing (after 5+ rounds): natural weaving of "What would make this fail?", "What shortcuts are tempting?", "What did a previous version get wrong?"
   - Domain-aware suggestions: based on detected project type, suggest common anti-patterns (SaaS, CLI, API examples)
   - Before writing NO-GOS.md: present ALL collected no-gos for user confirmation

5. **Add anti-pattern examples** to prevent interrogation mode:
   - BAD: "What about caching?" (out of nowhere)
   - GOOD: "You mentioned handling 10K concurrent users -- have you thought about caching strategy?"
   - Explicit instruction: "Follow the user's thread first. Only weave checklist domains when natural pauses occur or when the user's response opens a related domain."

6. **Expand `templates/templates/no-gos.md`** from 9 lines to a structured template with:
   - Header with Generated/Source metadata placeholders
   - "Hard Constraints" section (non-negotiable technical boundaries)
   - "Anti-Patterns" section (patterns to avoid with rationale)
   - "Previous Failures" section (lessons from past attempts)
   - "Domain-Specific Risks" section (common pitfalls for the project type)
   - Footer noting confirmation status

Do NOT add any runtime code or CLI commands for tracking. The AI manages all state in-context.
</action>
