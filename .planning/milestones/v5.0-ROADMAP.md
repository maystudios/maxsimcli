# Milestone v5.0: Context-Aware SDD

**Status:** SHIPPED 2026-03-08
**Phases:** 1-5
**Total Plans:** 14

## Overview

MAXSIM v5.0 focused on what MAXSIM didn't do yet: prevent its own context rot, ask deep enough questions during init, coordinate agents as a system, detect spec drift, and fill workflow gaps. Five phases, clean numbering, no legacy baggage.

## Phases

### Phase 1: Context Rot Prevention

**Goal**: MAXSIM actively prevents context accumulation in its own planning documents -- completed phases are archived, stale state is pruned, and active docs contain only current-milestone context
**Depends on**: Nothing (first phase)
**Requirements**: ROT-01, ROT-02, ROT-03, ROT-04
**Plans**: 2 plans in 2 waves
  - [x] 01-01: Phase archive sweep (ROT-01, ROT-03) — Wave 1
  - [x] 01-02: Stale detection + milestone reset (ROT-02, ROT-04) — Wave 2

**Success Criteria** (all met):
  1. Completing a milestone archives its phase directories to `.planning/archive/<milestone>/` and removes them from ROADMAP.md
  2. STATE.md is reset to current-milestone context on milestone transition
  3. A reprocess command detects planning documents with stale phase references and offers cleanup
  4. ROADMAP.md for a 50-phase project contains only the active milestone's phases

### Phase 2: Deep Init Questioning

**Goal**: New-project and init-existing produce context documents thorough enough that any agent can start work without asking follow-up questions
**Depends on**: Nothing (independent)
**Requirements**: INIT-01, INIT-02, INIT-03, INIT-04
**Plans**: 3 plans in 2 waves
  - [x] 02-01: Questioning depth + no-gos + research agents (INIT-01, INIT-02, INIT-03) — Wave 1
  - [x] 02-02: CONVENTIONS.md + PROJECT.md templates (INIT-04) — Wave 1
  - [x] 02-03: Workflow integration + CLI wiring (INIT-01, INIT-02, INIT-03, INIT-04) — Wave 2

**Success Criteria** (all met):
  1. Init flows ask explicit questions about tech stack, libraries, and tooling choices
  2. Both flows surface and document no-gos, hard constraints, and anti-patterns
  3. An agentic research step investigates tech stack choices and surfaces trade-offs
  4. A fresh agent receiving only PROJECT.md + REQUIREMENTS.md + NO-GOS.md can begin work without clarification

### Phase 3: Agent Coherence

**Goal**: Agents operate as a coordinated system -- prompts complement each other, context is role-targeted, and two-stage review is the default post-task workflow
**Depends on**: Phase 2
**Requirements**: AGENT-01, AGENT-02, AGENT-03, AGENT-04
**Plans**: 4 plans in 2 waves
  - [x] 03-01: Agent coherence - core agents (AGENT-01, AGENT-04) — Wave 1
  - [x] 03-02: Agent coherence - support agents + registry (AGENT-01, AGENT-04) — Wave 1
  - [x] 03-03: CLI init commands + frontmatter schema (AGENT-02) — Wave 2
  - [x] 03-04: Two-stage review enforcement (AGENT-03) — Wave 2

**Success Criteria** (all met):
  1. Each agent prompt explicitly references which other agents it hands off to
  2. Context assembly per agent role is defined (executor, reviewer, planner each get role-specific context)
  3. Two-stage review runs automatically after every task completion
  4. Agent handoff points documented and implemented -- no context loss between transitions

### Phase 4: Spec Drift Management

**Goal**: Users can detect and correct divergence between `.planning/` spec and actual codebase state using a single command
**Depends on**: Phase 3
**Requirements**: DRIFT-01, DRIFT-02, DRIFT-03, DRIFT-04
**Plans**: 3 plans in 2 waves
  - [x] 04-01: Drift core module + types + CLI wiring (DRIFT-01, DRIFT-02) — Wave 1
  - [x] 04-02: Drift-checker agent + check-drift command/workflow (DRIFT-01, DRIFT-02) — Wave 2
  - [x] 04-03: Realign command + workflow (DRIFT-03, DRIFT-04) — Wave 2

**Success Criteria** (all met):
  1. `/maxsim:check-drift` produces a severity-tiered drift report
  2. The drift report identifies specific mismatches with evidence per finding
  3. Realign-to-code path updates `.planning/` to reflect current codebase
  4. Realign-to-spec path generates fix phases for implementation gaps

### Phase 5: Workflow Coverage

**Goal**: Fill the two verified gaps in workflow coverage -- todo/bug discussion flow and phase listing pagination
**Depends on**: Nothing (independent)
**Requirements**: FLOW-01, FLOW-02
**Plans**: 2 plans in 1 wave
  - [x] 05-01: Discuss command and workflow (FLOW-01) — Wave 1
  - [x] 05-02: Phase listing pagination (FLOW-02) — Wave 1

**Success Criteria** (all met):
  1. `/maxsim:discuss` runs a collaborative discussion for small work items, triaging to todo or phase
  2. Phase listing supports pagination for 50+ phases with offset/limit

## Milestone Summary

**Key Decisions:**
- Clean slate planning rewrite -- archived all v4.x phases, renumbered from 1
- Context rot prevention as Phase 1 -- practice what we preach
- YOLO mode with balanced model profile for iterative development
- Universal two-stage review (spec + code) unconditional on all model profiles
- Drift-checker agent uses 5-pass multi-pass protocol to prevent context overload
- Realign workflow is an interactive orchestrator, not agent spawn

**Issues Resolved:**
- Context rot in MAXSIM's own planning documents (50+ phase accumulation)
- Shallow init questioning that missed constraints and no-gos
- Agents operating as isolated prompt islands without coordination
- No way to detect spec-code divergence
- Missing triage flow for small work items

**Issues Deferred:**
- Large module refactoring (server.ts, verify.ts, phase.ts) -- tech debt, not feature work
- Community skills marketplace
- Multi-user collaboration support

**Technical Debt Incurred:**
- Sync file I/O in CLI tool functions (acceptable for subprocess-per-call pattern)
- drift.ts conventions path checks `.planning/CONVENTIONS.md` but codebase-mapper writes to `.planning/codebase/CONVENTIONS.md`
- MCP mcp_list_phases reimplements pagination inline rather than reusing cmdPhasesList
