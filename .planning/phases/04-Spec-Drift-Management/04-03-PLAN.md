<objective>
Create the `/maxsim:realign` command and workflow that enables users to correct spec-code divergence in either direction: updating `.planning/` to match code (realign-to-code) or generating fix phases to make code match spec (realign-to-spec). This consumes the DRIFT-REPORT.md produced by `/maxsim:check-drift`.
</objective>

<execution_context>
Plan 01 (Wave 1) created the drift core module with init context and CLI tools. Plan 02 (Wave 2, parallel) creates the check-drift agent that produces DRIFT-REPORT.md. This plan creates the realignment workflow that reads the existing DRIFT-REPORT.md and acts on it.

The realign workflow is an interactive orchestrator (not an agent spawn) because it requires user decisions per item. It does NOT need the drift-checker agent -- it reads the already-written DRIFT-REPORT.md.

Key user decisions to honor:
- Realign-to-code: item-by-item approval (Accept/Skip/Edit per item)
- Realign-to-spec: generate new MAXSIM phases inserted after current phase
- All-or-nothing per direction (no cherry-picking individual mismatches)
- Both inline and standalone commands (check-drift offers realignment; /maxsim:realign reads latest report)
- Fix phases inserted after current phase
- Auto-mark phases complete if all criteria met
- Group related gaps into fewer phases (cap at 5, avoid phase explosion)
- Group by requirement prefix (e.g., all DRIFT-* together) or by affected subsystem
</execution_context>

<action>
Create `templates/workflows/realign.md` -- the orchestration workflow for realignment.

This is an interactive workflow (NOT an agent spawn) because it requires user decisions per item for realign-to-code and orchestrator-level phase creation for realign-to-spec.

**Step 1 -- Init:** Call `node ~/.claude/maxsim/bin/maxsim-tools.cjs init realign $DIRECTION` to get RealignContext. Parse the JSON. Validate that `has_report` is true. If not, tell user to run `/maxsim:check-drift` first and stop.

**Step 2 -- Read report:** Call `node ~/.claude/maxsim/bin/maxsim-tools.cjs drift read-report` to get the DRIFT-REPORT.md content and frontmatter. Parse the frontmatter to get status, counts. If status is "aligned", tell user "No drift detected -- nothing to realign" and stop.

**Step 3 -- Direction selection:** If direction was provided via $ARGUMENTS, use it. Otherwise, present the user with a choice:
- `to-code` -- "Update .planning/ to match current codebase (spec follows code)"
- `to-spec` -- "Generate fix phases to make code match spec (code follows spec)"

**Step 4A -- Realign to Code (if direction is to-code):**

Parse the drift report body to extract all drifted items (from "Spec Ahead of Code" and "Code Ahead of Spec" sections). For each item:

1. Present the item to the user: show the requirement ID, current spec text, what was found in code (or not found), and the report's fix recommendation.
2. Offer three choices: **Accept** (apply the spec change), **Skip** (leave as-is), **Edit** (user provides custom change).
3. If accepted:
   - For "Spec ahead of code" items (requirement marked done but code missing): update REQUIREMENTS.md to unmark the requirement (`[x]` -> `[ ]`). Also check and update ROADMAP.md success criteria if referenced. Also check STATE.md decisions if referenced.
   - For "Code ahead of spec" / undocumented items: add the feature to REQUIREMENTS.md as a new entry, or update PROJECT.md to document it -- based on user's edit.
   - For no-go violations: document as known tech debt in STATE.md or mark for future fix.
   - **CRITICAL: For each accepted item, identify ALL spec files that reference it and update all of them.** Use the spec file references from the drift report evidence section. This prevents internal spec inconsistency (Pitfall 5 from research).
4. After processing all items, check if any phase now has all success criteria met in code. If so, auto-mark that phase complete in ROADMAP.md using `node ~/.claude/maxsim/bin/maxsim-tools.cjs phase complete <phase>`.
5. Use `node ~/.claude/maxsim/bin/maxsim-tools.cjs requirements mark-complete <IDs>` for requirement status updates.
6. Commit changes: `node ~/.claude/maxsim/bin/maxsim-tools.cjs commit "docs: realign spec to match codebase" --files <changed-files>`

**Step 4B -- Realign to Spec (if direction is to-spec):**

Parse the drift report to extract all "Spec Ahead of Code" items (requirements/criteria that are specified but not implemented).

1. Group the gaps into phases using this algorithm:
   - First, group by requirement prefix (e.g., all ROT-* gaps into one phase, all INIT-* into another).
   - If no prefix grouping is possible, group by affected subsystem (cluster by common file paths in the evidence).
   - Cap at 5 phases maximum. If more than 5 groups, merge the smallest groups into a "Remaining Gaps" phase.
   - Each phase should have at minimum 2 gaps (unless it's the only group).

2. Present the proposed phase breakdown to the user. Show phase name, requirements included, and brief description. Wait for approval.

3. For each approved phase, insert it after the current active phase using `node ~/.claude/maxsim/bin/maxsim-tools.cjs phase insert <after_phase> "<name>"`. The insert command handles decimal numbering and ROADMAP.md updates.

4. For each inserted phase, write a basic goal and requirements list to the ROADMAP.md phase section. Include the requirement IDs and success criteria extracted from the drift report.

5. Commit changes: `node ~/.claude/maxsim/bin/maxsim-tools.cjs commit "feat: add realignment phases from drift report" --files .planning/ROADMAP.md <new-phase-dirs>`

**Step 5 -- Summary:** Display what was changed:
- For to-code: count of accepted/skipped/edited items, files modified, phases auto-completed
- For to-spec: count of new phases created, requirement IDs covered

**Error handling:**
- If DRIFT-REPORT.md is missing: direct user to run `/maxsim:check-drift` first
- If report status is "aligned": nothing to do
- If all items are skipped (to-code): no changes made, inform user
- If gap count is 0 (to-spec): inform user that all spec items appear implemented
